# Backlog

Read this index before opening detailed plans or proposing a new item. Keep entries ordered by intended consideration priority.

> **Cross-tool integration contract from PL-20260711 (Claude Code / VS Code)**: Every integration is exactly **two parts, at user scope**: (1) a **single native entry point** in the tool's user-level startup config that loads canonical `AGENTS.md` *content* — verify the tool's official native loading syntax first; prose that merely names or imperatively references an external file is **not** an import; (2) a **trusted source-directories grant** — the tool's narrowest persistent mechanism marking the Adaptive Agents repository safe to **read and write** from sessions rooted elsewhere (read for `INDEX.md` routing; write for routed workflows persisting retrospectives and planning artifacts back). Change only these narrow surfaces, preserve and deduplicate unrelated user configuration, and keep same-version reruns content-idempotent. Do not generate guidance copies, rule files, hooks, or skill markers unless verified tool behavior requires them. Validate the exact destination and effective config, then dogfood from an unrelated repository across fresh sessions with three probes: the sentinel (`ADAPTIVE_AGENTS_GLOBAL_LOADED`), a **content-proof probe** answerable only from repository content such as naming the current active plan (the sentinel alone can be a false positive — a stale installed copy can echo it, as PL-20260711 OpenCode proved), and a routed write-back (e.g., retrospective capture). Intermittent loading is a failure, not a pass.
>
> Additional installer duties (from the PL-20260711 OpenCode rework): **Migrate, don't accumulate** — each installer detects and removes its own prior-generation artifacts, including writes to previously wrong destinations, matching installer-generated signatures only and never deleting user-authored files. **Isolated automated tests** — each installer ships a test script running entirely against temp-dir config (fresh install, legacy migration, byte-stable rerun, dry-run writes nothing, unrelated-config and user-file preservation); it must never touch the real user configuration. **Live health check** — each integration adds a contract-validation function to `scripts/check-adaptive-agents.sh` that parses the tool's real config (string-aware JSONC parsing; naive comment-stripping corrupts `https://` URLs). **Version pinning** — record the tool version against which docs and behavior were verified in the work-unit memory, and re-verify on major tool upgrades before trusting prior findings.

## Epic / Child Conventions

When a backlog item grows too large for a single activation (its scope spans multiple independent deliverables), split it into an epic with children:

- **Epic**: A backlog item with `Status: Epic` and `Type: Epic`. Holds the high-level Objective, Problem Spec, architecture overview, and resolved decisions. Never activated directly — only children are activated.
- **Children**: Independent backlog items living in a subdirectory named after the epic (`PL-YYYYMMDD-slug/PL-YYYYMMDDX-slug.md`). Each has its own Objective, Scope, AC, and Readiness.
- **Dependencies**: Children declare `Depends on` pointing to sibling files for ordering. Activate only when dependencies are `Completed` or `Ready`.
- **Backlog index**: The epic appears as a single row in INDEX.md; children are indented beneath it. This keeps the index scannable while making the hierarchy visible.
- **Directory structure**: The epic file lives at `backlog/PL-YYYYMMDD-slug.md` and children live in `backlog/PL-YYYYMMDD-slug/PL-YYYYMMDDX-slug.md`. This prevents naming collisions and makes the relationship discoverable without INDEX.md parsing.
